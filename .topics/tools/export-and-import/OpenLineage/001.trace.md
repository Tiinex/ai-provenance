# Evidence: OpenLineage source-model slice for export and import

## Provenance

- Source: OpenLineage object-model documentation and the preserved ChatGPT platform suggestion on lineage schemas
- Origin:
  - [relative](../../validator/001-3-message-from-chatgpt-platform.trace.md)
  - [absolute](C:/Users/micro/Documents/Repos/Tiinex/ai-provenance/.topics/tools/validator/001-3-message-from-chatgpt-platform.trace.md)
  - [OpenLineage object model](https://openlineage.io/docs/spec/object-model)
- Representation: bounded evidence note that keeps the source model and the Tiinex export and import question separate

## Evidence Material

- OpenLineage models lineage as a graph of Jobs, Runs, and Datasets.
- Run state updates are additive across a run lifecycle and carry timestamps.
- Jobs and Datasets are identified by names and namespaces.
- Facets carry extensible metadata for runs, jobs, datasets, inputs, and outputs.
- Schema URLs and versioned pointers are used for extensible metadata and stable references.
- The object model is rich enough to act as a source format for a Tiinex export adapter.

## Supports

- Claim: OpenLineage is a source lineage model, not a Tiinex schema note.
- Claim: Tiinex export and import needs an explicit mapping layer for identity, lifecycle, and facets.
- Claim: source-model fidelity matters if `.trace.md` exports are expected to round-trip or interoperate later.

## Interpretation Notes and Limits

- This note is about the source model and the mapping pressure it creates, not a final Tiinex adapter spec.
- The OpenLineage docs are used as context for export and import design, but they are not themselves Tiinex canon.
- The ChatGPT suggestion remains design input rather than a ratified format rule.
