# Tools Program Root

This topic is the umbrella for the ai-provenance tools line.

## Current Read

The current tools work is not one project. It is three distinct operating
fronts that need different shapes:

- validator: machine findings and trust in what the validator says
- repair: bounded rewrite and recovery after a finding is real
- ux: human-readable surfaces that make the same trace state understandable

The trace-format/tools lineage already proved that tooling grows better when it
is split into concrete tasks instead of staying as one broad tools cloud. This
root carries that lesson into the broader ai-provenance tools space.

For code-change evidence, a committed change-set hash plus one top-level
`.trace.md` note is enough to find the box again; per-file evidence only earns
its own leaf when the file itself carries a distinct state, proof, or repair
boundary.

## Design Direction

Current tools work should prefer:

- validator output first, because repair should not outrun trustworthy findings
- repair proposals only after the evidence is strong enough to inspect
- viewer surfaces that expose the same trace truth in readable chat and
  detailed modes
- small, named branches rather than one giant tools umbrella with no working
  slices

## Current Branches

- [validator/001.trace.md](validator/001.trace.md) - machine finding and validation lane
- [repair/001.trace.md](repair/001.trace.md) - bounded repair lane
- [ux/001.trace.md](ux/001.trace.md) - UX lane for viewer and future operator surfaces

## Next Artifacts

- capture the code-change provenance and evidence-granularity decision as its
  own child leaf under this root
- continue validator with the false-positive and Problems-policy follow-ups
- define the repair branch as audit/proposal/apply rather than a vague fix-all
  tool
- keep the UX branch centered on the viewer surface before adding any
