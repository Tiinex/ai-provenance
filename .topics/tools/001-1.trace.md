# Code-Change Provenance Granularity

## Decision

- State: accepted
- Subject: how code-change provenance should be carried for ai-provenance tools work
- Decision: use one committed change-set hash and one top-level `.trace.md` note as the normal provenance anchor for a code change-set

## Basis

- the `.trace.md` note sits beside the code change and describes the moment
- the commit hash is the box label that lets the archive be found again
- per-file evidence is useful only when a file has its own distinct state, proof, or repair boundary

## Consequences

- code-change evidence stays compact instead of bloating into one artifact per changed file
- descendants can inherit the rule when they refresh their own checksums after a root-level policy update
- more granular evidence remains available for files that genuinely carry separate state

## Operational Effect

- this decision applies to new ai-provenance tooling change-sets that are being documented through the traced lineage
- use file-level evidence only when the file itself is the thing being proven, not just when it was edited
