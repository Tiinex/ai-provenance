# Repair Tools Root

This topic tracks the repair lane as its own bounded tools surface.

## Current Read

Repair should live between finding and mutation.

The validator tells us what is broken. The repair lane explains what a safe rewrite would change, and it stops when the evidence is weak or the blast radius is too broad.

This branch should inherit the existing schema-lineage repair discipline rather than reinvent it.

## Design Direction

The repair branch should keep these distinctions visible:

- audit findings: classification only
- repair proposals: specific before/after targets
- applied repair: confirmation that the rewrite actually landed
- lineage semantics: what relation is intended
- carrier semantics: how Git or another carrier stores and repins that relation

## Current Scope

The repair branch should eventually cover:

- stale or weak parent targets
- wrong published parents
- digest refresh after legitimate rewrites
- origin upgrade from weak local path to committed browseable target when the evidence supports it

## Boundaries

The repair branch should not yet try to:

- auto-fix every broken edge
- choose between ambiguous candidates silently
- hide repair history
- blur the difference between a validator finding and a repair action

## Next Artifacts

- an audit-finding note for the first repairable failure class
- a proposal-preview note showing a concrete before/after rewrite
- an applied-repair note that records what changed and why
