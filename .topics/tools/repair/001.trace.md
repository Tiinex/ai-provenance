# Repair Root

This topic tracks repair discipline as its own bounded line of work.

The immediate goal is not to automate every repair path.

The immediate goal is to make the repair model explicit enough that later
tooling does not confuse Tiinex lineage semantics with the behavior of one
particular storage carrier.

## Current Read

The recent checksum and parent-link work exposed an important distinction:

- Tiinex lineage rules describe semantic relations such as parent, origin,
  evidence, decision, and continuity
- Git-grounded repair imposes an extra dependency order because commit-pinned
  references and content-bound identities rotate when content changes
- other carriers may not rotate identities in the same way, so the same repair
  discipline should not be treated as universally inherent to lineage itself

That means current repair guidance should be written in layers rather than as
one blended rule set.

## Layer Split

For this topic, repair work should distinguish at least these layers:

1. lineage semantics
2. carrier semantics
3. repair discipline

Current working meanings:

- lineage semantics: what relations the artifact intends to preserve
- carrier semantics: how the underlying storage or versioning system treats
  identity, mutability, and history
- repair discipline: the concrete order of edits, checksum updates, and commits
  required to keep the lineage truthful under that carrier

## Git-Grounded Repair Read

For Git-grounded lineage, current repair discipline should assume:

1. a parent should stabilize before a dependent child is repaired against it
2. a child that stores a parent checksum should be refreshed only after the
   parent relation it points to is settled enough for that step
3. commit-pinned cross-repo origin should be treated as a stable grounding
   reference, not as a requirement to mirror the latest parent commit on every
   later edit
4. dependency order matters when several traces depend on each other through
   content-bound identity

This is a Git-grounded discipline, not a universal law of lineage.

## Non-Git Read

Other carriers may behave differently.

Examples of differences that matter:

- stable object ids that persist across metadata changes
- explicit version layers beneath one permanent object identity
- append-only relation logs that add links without mutating older identifiers

When a carrier behaves that way, the repair order may be less rotational than
the current Git-grounded case.

## Design Direction

Current repair tooling and guidance should therefore prefer:

- telling the user or tool which part is a lineage rule versus a Git rule
- avoiding language that makes Git-specific sequencing sound like a universal
  Tiinex invariant
- treating commit order as carrier discipline rather than as the full meaning
  of continuity itself

## Next Artifacts

- add a short Git-grounded repair protocol when the current discipline needs a
  more operational checklist
- add a later comparison note if another carrier is tested and exposes a
  materially different repair dynamic
