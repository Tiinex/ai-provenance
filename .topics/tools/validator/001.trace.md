# Validator Root

This topic tracks validator work as its own bounded line of progress.

The immediate goal is not to make every current artifact green.

The immediate goal is to make the validator trustworthy enough that a reported
finding is worth human attention.

## Current Read

The current validator now exposes useful machine-found findings, but the recent
work also showed how easy it is to drift into repairing individual artifacts
before the validator's own signal policy is stable.

That is the wrong order for this track.

For this topic, validator quality comes first.

Recent progress on this root concern is now concrete:

- the validator core now emits a normalized `findings` surface with stable
  finding codes, categories, messages, severities, and default human surfaces
- Problems diagnostics now read that machine finding surface instead of
  rebuilding their own local status-to-message mapping

This reduces drift between the bounded validation report and the direct editor
surface.

## Working Order

The working order for this validator track is:

1. validator surfaces findings
2. we verify whether the findings are real
3. we remove false positives from the validator
4. only after false positives are reduced do we interpret what is actually
   missing from the artifact set

This order keeps validator defects separate from content defects.

## Decision Read

For this topic, the validator should be treated as a machine-first proof and
triage surface.

That means:

- validator output should be deterministic where possible
- the validator should prefer explicit status classes over fuzzy judgment
- human-facing surfaces may summarize findings, but they should not invent the
  underlying machine result
- repository cleanup should not be used to hide validator mistakes

## Validator Priorities

The current priority order is:

1. make sure the validator can surface concrete failures
2. classify whether each surfaced failure is genuine, stale, legacy, or false
   positive
3. tune the validator until false positives are acceptably low
4. only then widen missing-signal or completeness analysis

## What Counts As A Good Validator Finding

A good validator finding should satisfy most of these properties:

- the finding is reproducible by rerunning the same validator logic
- the finding points at a concrete local field, link, checksum, or structural
  expectation
- the validator can explain the class of failure without needing an LLM to
  guess what happened
- the finding is actionable enough that a human can decide whether the problem
  is in the validator or in the artifact

## What Counts As A False Positive For This Track

For this topic, a false positive includes at least these cases:

- the validator flags an artifact because it belongs to a legacy or draft era
  that the current auto-diagnostics policy should not treat as actionable noise
- the validator reports a failure class that the current parser actually
  misunderstood
- the validator duplicates or inflates one defect into several misleading
  human-facing problems
- the validator treats intentionally absent proof as if it were the same thing
  as a broken proof

## Current Scope

This topic currently covers:

- continuity footer checks
- direct-parent checksum checks
- backward parent traversal
- runtime-trace structural checks
- human-facing diagnostics policy for which findings should appear directly in
  Problems versus which should stay in bounded validation reports

## Current Non-Goals

This topic is not yet trying to:

- auto-repair the full repository
- settle every historical artifact format
- flatten legacy, draft, and canonical artifacts into one identical rule set
- use LLM judgment as the source of truth for validator pass or fail

## Next Artifacts

This root should likely grow at least these companion notes:

- a finding taxonomy for validator output classes
- a false-positive log with short repro notes
- a policy note for what should and should not appear in Problems
