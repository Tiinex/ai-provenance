# Trace Format Redesign Root

This topic captures the current design direction for `ai-provenance`
trace-format work.

It continues from the current RFC envelope draft rather than standing as an
isolated local lineage root.

It also intentionally does not inherit older `.topics` structure assumptions
as canon. Some older topic material may still match current implementation,
but it no longer reflects the full lineage of the design dialogue that now
drives the next format update.

## Current Read

The current implementation still stores parent trace links primarily as
relative or absolute filesystem paths.

That is sufficient for many local cases, but it is too weak for stable
cross-origin continuity when the parent trace lives in another repository.

The design direction now grounded in the latest dialogue is:

1. Same-repo `.trace.md` parents should always normalize to relative paths.
2. Cross-repo `.trace.md` parents should default to protocol-bearing origin
   locators rather than pretending relative paths remain stable.
3. The first origin driver should be a git driver.
4. Cross-origin parent traces should default to requiring a committed target.
5. A temporary override may permit weaker relative links, but that mode is a
   deliberate integrity downgrade rather than the normative path.

## Header Direction

The continuity header currently being shaped is expected to support at least:

- `Parent Trace`
- `Parent Origin`
- `Parent Created At`
- `Created At`
- `Why`
- `Summary`
- `Type`
- `Schema`

Working intent for those fields:

- `Parent Trace` remains the direct continuity parent reference.
- `Parent Origin` carries the adapter-selecting locator when inherited origin
  cannot safely remain local or when the parent lives in another repo.
- `Parent Created At` records the known creation time of the parent trace when
  that relationship is present.
- `Created At` records when the current trace artifact was created using the
  `YYYY-MM-DD hh:mm:ss` shape without milliseconds.
- `Why` explains why the trace exists.
- `Summary` acts as an artifact-level fallback description similar to an `alt`
  description rather than a new truth source.

## Cross-Origin Direction

The active design direction is not to replace all path logic with origin
drivers.

Instead, the repo should keep ordinary relative-or-absolute behavior for
non-trace paths and for same-repo trace relationships, while introducing a
separate stronger path for cross-origin trace references.

That means:

- path policy remains useful
- origin drivers become an additional capability
- commit-pinned cross-origin references become the default strong form
- override remains possible for recovery or repo-move scenarios

The same direction should extend beyond parents. References should be treated
as driver-reachable destinations with attached timestamps rather than as a
special case limited to ordinary file paths.
