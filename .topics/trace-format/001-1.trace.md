# Selective Readability Decision

## Decision

- State: accepted for current trace-format direction
- Subject: how to evaluate markdown lineage versus JSON lineage for LLM reading without custom tooling
- Decision: markdown lineage should be treated as cheaper than JSON lineage when the artifact shape lets a reader reliably skip low-signal wrapping and jump directly to schema, summary, decision, parent, and other high-signal sections

## Basis

- Raw character count is not the same thing as inference cost for an LLM.
- JSON often carries uniform wrapping that is easy to parse mechanically but not automatically cheap to interpret semantically.
- Tiinex lineage markdown already exposes several strong skip anchors such as `Current`, `Summary`, `Why`, `Decision`, and parent references.
- The practical cost driver is whether a reader can tell what matters quickly, not whether the transport syntax uses braces or headings.

## Consequences

- Trace-format work should optimize for selective readability rather than assuming a more machine-shaped syntax is automatically cheaper.
- Future lineage fields should make high-signal sections easier to detect and easier to distinguish from explanatory padding.
- A format change toward more JSON-like wrapping should need a stronger justification than "it looks more structured".

## Review Conditions

- Revisit if later custom tooling changes the real traversal cost enough that selective human-readable structure matters less.
- Revisit if repeated lineage reading shows that current markdown sections still force too much broad rereading despite the intended skip anchors.
- Revisit if a hybrid surface proves cheaper in practice without flattening semantic signal.
